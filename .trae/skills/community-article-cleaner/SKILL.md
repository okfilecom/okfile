# Community Article Cleaner

用于把原始文章整理成更适合开发者社区发布的“社区版”内容，重点处理去广告化、去引流化、脱敏，以及标题、摘要、开头、结尾的重写。

适用场景：

- 用户要求把原始 Markdown、宣传稿、博客草稿改成社区版
- 用户要求把文章从“产品介绍”改成“技术记录”“测试报告”“工程复盘”
- 用户准备把同一篇文章投递到多个开发者社区
- 文章被判定为广告、推广或引流，需要先改写再重提

## 目标

输出一份可直接交给发布类 skill 或人工发布的社区版材料，通常包括：

- 社区版标题
- 社区版摘要
- 社区版正文
- 可选标签建议
- 若有必要，附带“删掉了什么、为什么删”的说明

## 推荐流程

1. 先读取原文，判断它更像技术记录、宣传稿还是混合稿。
2. 标记所有高风险内容：
   - 推广措辞
   - 号召性文案
   - 聚合外链
   - 未脱敏凭证
   - 不适合公开的内部信息
3. 重写标题、摘要、开头、结尾，再统一收紧正文语气。
4. 保留技术事实、步骤、命令、现象和结论，删减营销信息。
5. 输出一版适合社区发帖的正文，必要时给出多个标题备选。

## 改写原则

优先把文章改成下面这些风格：

- 技术记录
- 测试报告
- 兼容性验证
- 性能基线
- 接口调试笔记
- 工程复盘

优先保留这些内容：

- 背景和测试目标
- 环境准备
- API 调用流程
- CLI 命令
- 静态站点发布步骤
- 观察结果和技术结论

优先删减这些内容：

- 明显推广句式
- “值得一试”“强烈推荐”“Give it a try”这类营销表达
- 直接分享链接、下载链接、注册链接
- “Free”“Perfect for”这类强营销措辞
- 官网、GitHub、PyPI 等聚合外链列表
- 未脱敏的 Key、Token、Cookie、Session

## 标题与摘要

- 标题优先写“做了什么验证/记录了什么问题”，不要写成产品宣传语
- 摘要优先写“本文记录了什么”，避免“帮助你快速上手”这类引导式表达
- 若原文品牌信息过重，可降级到中性表述，只保留必要上下文

## 正文重写

- 开头直接讲问题背景、测试目标或约束条件
- 主体部分按步骤、实验过程、现象、结论组织
- 结尾写工程判断、适用边界、后续待验证项
- 不要在结尾使用“欢迎试用”“立即体验”之类 CTA

## 脱敏要求

- 不公开 API Key、Access Token、Cookie、Session、邀请码
- 示例命令和请求体里的敏感值统一替换为占位符
- 若原文含真实账号、内部域名、工单号，按需泛化或删除

## 输出建议

默认至少输出以下结构，便于后续复用：

- `标题`
- `摘要`
- `正文`
- `可保留的技术标签`
- `需注意的平台风险`

如果用户明确要求多平台复用，可再补一段：

- `可通用版`
- `偏腾讯云/阿里云的社区版`
- `偏 Reddit 的轻量版`

## 与发布 skill 的协作

- 本 skill 只负责内容清洗和改写，不负责实际点按钮发帖
- 内容整理完成后，再交给发布类 skill 执行浏览器自动化
- 若平台已驳回，优先重写标题、摘要、首段和结论，再进入重提流程
