# 如何通过 OkFile 实现文件的阅后即焚

很多时候，我们分享文件并不是想让它长期存在，而是只希望对方看一次、下载一次，随后链接立即失效。

这类场景很常见：

- 发送一次性报价单、截图、账单或内部文档
- 给协作者临时看一个调试文件，不希望链接长期可访问
- 给 Agent 或自动化流程传递一次性输入，避免后续重复读取
- 降低“链接被转发后长期裸奔”的风险

如果只是把文件上传到普通对象存储，通常只能做到“有链接就一直能访问”，除非你额外写一套过期控制、下载计数和删除逻辑。

`OkFile` 现在已经把这件事直接做进了上传流程里。你只需要在上传时打开 `burnAfterRead`，就可以把一个普通文件变成“一次性链接”。

这篇文章会讲清楚几件事：

1. `OkFile` 的阅后即焚到底是什么行为
2. 如何通过页面、CLI 和 API 三种方式开启它
3. 它对直链、预览页和下载页分别怎么生效
4. 这个能力在实现上踩过哪些坑

---

## 一、先说结论

如果你只想快速上手，可以直接记住这一条：

- 上传文件时加上 `burnAfterRead=true`

上传完成后返回的这些链接：

- `url`
- `downloadUrl`
- `playUrl`

都会被纳入同一套“一次性消费”逻辑。

也就是说，只要有人第一次成功读取到文件内容，这个文件就会立即失效，后续访问会返回失效页，而不是继续拿到原文件。

当前行为可以概括为：

- `GET /i/{id}`：第一次成功读取后失效
- `GET /d/{id}`：第一次成功下载后失效
- `GET /i/{id}?play=1`：第一次成功打开预览页后失效

后续访问会返回 HTTP `410 Gone` 风格的过期页面，而不是继续返回原始内容。

---

## 二、什么叫“阅后即焚”

很多服务声称支持“阅后即焚”，但实际行为并不一致。

有的做法只是：

- 设置一个较短的时间过期
- 或者限制下载次数
- 或者只是隐藏链接入口

`OkFile` 这里的定义更直接：

> 只要第一次成功预览或下载到内容，这个文件对象和相关公开链接就立即失效。

这和“10 分钟后过期”是两回事。

因为“时间过期”关注的是多久之后不可访问，而“阅后即焚”关注的是：

> 有没有人真的读到它

这对敏感文件尤其重要。你往往不是担心它 10 分钟后是否还活着，而是担心：

- 对方看完之后还能不能继续转发
- 浏览器缓存会不会让它继续被打开
- 预览页是不是还能反复访问

这几个问题，正是 `OkFile` 这次实现里重点处理的部分。

## 三、通过上传页面开启阅后即焚

如果你不想写命令或 API，最简单的方式就是直接用上传页。

入口：

- `https://www.okfile.com/en/upload/`
- `https://www.okfile.com/zh/upload/`

上传页现在已经有独立的 `Burn after read` 选项。

使用方式：

1. 选择一个文件
2. 勾选 `Burn after read`
3. 开始上传
4. 拿到返回的链接

页面里会明确提示这是一个一次性文件，避免误以为它仍是普通公开链接。

需要注意的是：

- 这个开关只对“文件上传”生效
- 对“站点发布”不生效

也就是说，如果你是发布一个完整目录到子域名，这个功能当前不会套到整站上。

---

## 四、通过 CLI 开启阅后即焚

如果你习惯终端或脚本化流程，CLI 是最直接的入口。

### 1. Python CLI

```bash
okfile upload secret.pdf --burn-after-read
```

或者：

```bash
py -3 -m okfile_cli upload secret.pdf --burn-after-read
```

### 2. npm 入口

如果你走 Node 侧的启动方式，也可以这样调用：

```bash
npx @okfilecom/okfile upload ./secret.pdf --burn-after-read
```

上传成功后，CLI 会返回：

- `URL`
- `Download URL`
- `Preview URL`

其中只要任意一个链接第一次成功拿到内容，其它相关链接也会一起失效。

这点非常重要。

它不是“每个链接单独算一次”，而是“这个文件只允许成功读一次”。

---

## 五、通过 API 开启阅后即焚

如果你是在给 Agent、脚本或服务端接入，推荐直接用 API。

`OkFile` 支持两条文件上传路径：

- `quick upload`
- `prepare -> upload -> complete`

两条路径都支持 `burnAfterRead`。

### 1. Quick Upload

适合小文件直接上传：

```bash
curl -X POST "https://www.okfile.com/api/upload/quick" \
  -F "file=@secret.pdf" \
  -F "burnAfterRead=true"
```

成功后返回类似：

```json
{
  "success": true,
  "id": "a3k7m92x",
  "url": "https://ok26.org/i/a3k7m92x",
  "downloadUrl": "https://ok26.org/d/a3k7m92x",
  "playUrl": "https://ok26.org/i/a3k7m92x?play=1",
  "type": "pdf",
  "burnAfterRead": true
}
```

### 2. Prepare Upload

适合大文件或需要自己控制上传过程的场景：

```bash
curl -X POST "https://www.okfile.com/api/upload/prepare" \
  -H "Content-Type: application/json" \
  --data '{
    "filename": "secret.pdf",
    "size": 12345,
    "contentType": "application/pdf",
    "burnAfterRead": true
  }'
```

后续再按正常流程：

1. 上传文件体到签名 URL
2. 调用 `POST /api/upload/complete`

完成后，这个文件同样会进入一次性访问模式。

## 六、三个链接分别怎么生效

上传完成后，`OkFile` 通常会给出三类链接：

### 1. `url`

形如：

```text
https://ok26.org/i/{id}
```

这是直接文件访问入口。

在开启 `burnAfterRead` 后：

- 第一次成功读取文件内容，立即作废
- 第二次访问就会进入失效页

### 2. `downloadUrl`

形如：

```text
https://ok26.org/d/{id}
```

这是受控下载入口。

开启 `burnAfterRead` 后：

- 第一次成功下载后立即失效
- 后续再次下载会返回过期页

### 3. `playUrl`

形如：

```text
https://ok26.org/i/{id}?play=1
```

这是图片、视频、PDF 等支持预览的页面入口。

这里曾经最容易出问题，因为“预览页”本身和“文件内容”并不是同一个响应。

如果只让 HTML 壳页加载出来，而不控制它里面实际请求的媒体地址，就会出现一种假象：

- 页面打开了
- 文件其实被缓存了
- 后面还能继续反复预览

`OkFile` 现在针对这一点做了额外处理：

- 预览页只会发出一次性媒体令牌
- 第一次成功打开预览页后，后续再次打开就直接失效
- 预览媒体响应也会带 `no-store`，避免浏览器缓存重放

所以 `playUrl` 现在也已经纳入真正的一次性消费语义，而不是“只是一个页面壳”。

## 七、这个能力解决了哪些真实问题

### 1. 链接被转发后的持续暴露

普通公开链接一旦发出去，就很难控制后续传播。

而阅后即焚至少可以把暴露窗口压缩到：

- 第一个成功访问之前
- 或第一次成功访问当下

### 2. 浏览器缓存导致“失效不彻底”

这是一次性链接里最容易被忽略的问题。

如果后端已经删除了对象，但浏览器本地还缓存着首个成功响应，用户仍然会觉得“它还能打开”。

这次实现里专门补了：

- `Cache-Control: private, no-store, max-age=0`
- `Pragma: no-cache`
- `Expires: 0`

目的就是尽量避免浏览器或中间层把第一次成功访问缓存下来。

### 3. 预览页和直链语义不一致

另一个常见坑是：

- `/i/{id}` 走的是直链
- `/i/{id}?play=1` 走的是预览页

如果两边没有统一的一次性消费语义，用户就会发现：

- 直链失效了
- 但预览页还能进

这会严重削弱“阅后即焚”的可信度。

所以 `OkFile` 现在已经把直链、下载页、预览页统一到了同一套失效模型里。

## 八、什么时候应该用它，什么时候不该用它

适合开启阅后即焚的场景：

- 临时共享敏感文档
- 一次性验证码截图、账单、报价单
- 给外部协作者临时看文件
- 给 Agent 传一次性输入文件
- 你明确不希望链接在首次访问后继续存活

不太适合开启的场景：

- 公开分发文件
- 文档、图片、视频需要多人重复访问
- 希望一个链接在一段时间内被反复打开
- 需要给多个接收方同时访问同一文件

如果你的目标是“可控访问，但不是一次性”，更适合考虑：

- `expiresAt`
- `maxDownloads`

它们和 `burnAfterRead` 是不同维度的控制方式。

## 九、一个最小可用示例

如果你今天就想试一下，最短路径就是：

### 页面方式

1. 打开 `https://www.okfile.com/en/upload/`
2. 选择文件
3. 勾选 `Burn after read`
4. 上传
5. 用返回的 `url` 或 `playUrl` 打开一次
6. 再打开第二次，应该直接失效

### CLI 方式

```bash
okfile upload secret.pdf --burn-after-read
```

### API 方式

```bash
curl -X POST "https://www.okfile.com/api/upload/quick" \
  -F "file=@secret.pdf" \
  -F "burnAfterRead=true"
```

## 十、最后总结

如果把文件上传理解成“存储”，那阅后即焚只是附加选项。

但如果把文件上传理解成“受控交付”，那这个能力其实非常关键。

`OkFile` 这次做的不是简单加一个布尔参数，而是把下面这些问题一起补齐了：

- 上传接口如何声明一次性语义
- CLI 和上传页如何暴露这个能力
- 直链、下载页、预览页如何统一失效
- 浏览器缓存如何避免“死而复活”
- 预览页如何避免只失效文件、不失效页面壳

所以现在你可以比较明确地把它当成真正可用的一次性文件链接，而不是“看起来像一次性”的实验功能。
