# 我是怎么用 WorkBuddy 和 Agent 把一个文件或静态网站发布出去的

前段时间我在整理一个 Agent 工作流时，卡在了最后一步。

前面的内容生成其实都不难。无论是让 Agent 生成一张图片、一份 PDF 报告，还是导出一个静态站点目录，真正麻烦的是最后那一下: 怎么把结果稳定地发布出去，并且给别人一个能直接打开的链接。

如果只是自己本地看，事情到这里就结束了。但只要进入协作场景，问题就会马上出现:

- Agent 生成的文件怎么上传
- 上传后返回什么链接最合适
- 如果是图片、视频、PDF，能不能直接预览
- 如果是整个静态目录，能不能直接当成网站访问
- 如果后面还要更新同一个站点，怎么避免线上半更新状态

我后来把这件事拆成了两部分:

- `WorkBuddy` 负责把任务组织起来，让人和 Agent 在一个流程里协作
- `Agent` 负责生成内容，并调用发布接口把结果交付出去

这样以后，无论生成的是一个文件，还是一个完整静态网站，最后都可以落到同一条发布链路上。

## 一、先说结论：问题不在“生成”，而在“交付”

很多时候我们会以为，Agent 能把内容生成出来，事情就完成了。

其实不是。

真正影响体验的，是交付动作有没有闭环。

比如下面这些场景：

- Agent 生成一份日报 PDF，需要立刻发给同事
- Agent 生成一张海报，需要给运营一个能直接打开的图片链接
- Agent 导出一个文档站点目录，需要给用户一个网站入口
- Agent 每天生成一版报告站点，需要持续更新同一个地址

这些都不只是“生成文件”，而是“生成后立即可访问”。

所以我后来在 WorkBuddy 里做任务设计时，会把发布也视为主流程的一部分，而不是一个临时补丁。

## 二、我用的基本思路

我的做法很简单:

1. 在 WorkBuddy 里定义任务目标
2. 让 Agent 生成最终产物
3. 根据产物类型，走文件发布或站点发布流程
4. 把返回的公开链接写回任务结果
5. 需要更新时，继续复用原有站点

这里有个关键点：不要让 Agent 只负责“产出文件”，而是让它对“可交付结果”负责。

也就是说，任务最终返回的最好不是“本地生成成功”，而是：

- 文件链接
- 预览链接
- 下载链接
- 站点入口

这样 WorkBuddy 里的任务结果才真正可消费。

## 三、场景一：让 Agent 发布单个文件

这是最常见的场景。

比如让 Agent 生成：

- PNG 图片
- MP4 视频
- PDF 报告
- Markdown 导出的文档
- CSV 或日志文件

这个流程我现在通常会这么设计：

1. Agent 在本地生成文件
2. 调用上传准备接口拿到上载地址
3. 把文件内容传上去
4. 调用完成接口
5. 获取最终返回的公开链接

如果是图片、视频、PDF，一般还会顺手拿到一个可直接打开的预览链接。

对 WorkBuddy 来说，这一步很重要，因为很多任务最后并不是要一个“文件对象”，而是要一个“我现在就能点开的结果”。

比如一个海报生成任务，如果最后只告诉你“图片已保存在本地目录”，这个任务其实还没真正结束。

但如果最后返回的是：

- 图片公开链接
- 在线预览链接

那这个任务就真的可以交付了。

## 四、场景二：让 Agent 发布一个静态网站

第二类更有意思：Agent 不只生成单文件，而是直接生成一整个目录。

我最近更常用的是这种场景：

- 生成日报站点
- 导出帮助文档
- 输出一个小型展示页
- 把分析结果整理成 HTML 报告

这时我不会让 Agent 把目录压缩成 zip 再交给人手动处理，而是直接让它发布整个目录。

一个比较自然的流程是：

1. Agent 输出站点目录
2. 调用站点准备接口
3. 上传目录中的所有文件
4. 调用站点完成接口
5. 返回站点入口链接

这样，WorkBuddy 中的任务结果就不再是“已生成目录”，而是“站点已可访问”。

这中间有几个很实用的点。

第一，目录结构要保留。

很多静态内容并不是一个 `index.html` 加几个零散文件，而是会带着：

- `assets/`
- `images/`
- `docs/`
- `scripts/`

如果发布时目录结构丢了，站点大概率就坏了。

第二，不要强迫 Agent 为了目录浏览去额外生成假首页。

有些导出结果本来就不是一个标准官网，而是一个资料目录、图片目录或者报告目录。根目录没有 `index.html` 并不奇怪。

这种情况下，更合理的做法是：

- 如果根目录存在 `index.html`，就直接渲染它
- 如果不存在，就展示目录列表

这样 Agent 只需要上传真实结构，而不是为了适配发布系统再多做一层伪造。

## 五、为什么 WorkBuddy 和 Agent 的组合很适合做这件事

我觉得这件事的关键，不只是“能不能上传”，而是谁来负责发布决策。

如果全部靠人手动做，流程会变成：

- Agent 生成结果
- 人下载结果
- 人找地方上传
- 人复制链接
- 人再把链接贴回任务系统

这个过程不但慢，而且很容易断链。

而 WorkBuddy 和 Agent 配合之后，发布动作可以被收进任务流里：

- WorkBuddy 给出任务目标和上下文
- Agent 判断输出类型
- Agent 自动选择文件发布或站点发布
- Agent 把最终链接回填到任务结果

对日常使用来说，这种差别非常大。

你看到的不再是“某个本地路径下生成了文件”，而是“这个任务已经交付完成，可以打开”。

## 六、我在实际使用里踩过的几个坑

如果你也想做这条链路，我建议提前注意下面这些事。

### 1. 不要把上传当成最后临时补的一步

最容易出问题的地方，就是前面把生成流程设计得很完整，最后才临时想“找个地方传一下”。

这样通常会导致：

- 返回值不统一
- 权限模型混乱
- 文件和站点两种场景分裂
- 后续更新没有版本概念

更好的方式，是从一开始就把“交付结果是什么”定义清楚。

### 2. 大文件不要只靠一次直传

如果 Agent 会生成较大的视频、压缩包或高分辨率图片，分片上传几乎是必须的。

否则一次失败就得整包重来，对自动化任务非常不友好。

### 3. 发布站点时要保留真实目录

不要在 Agent 侧擅自改写目录结构，也不要把一整个目录硬塞进某个单文件字段里。

静态站点能否正常打开，很多时候依赖的就是原始相对路径。

### 4. 更新已有站点时要避免“覆盖一半”

这个坑非常典型。

如果一个已上线站点直接原地覆盖，线上很可能短时间出现混合状态：

- HTML 已经是新版
- CSS 还是旧版
- 图片还没传完

所以我后来会优先采用“先生成新版本，完成后再切换”的方式。

这样至少不会把一个更新到一半的站点暴露给用户。

## 七、一个我觉得比较顺手的任务模板

如果你想在 WorkBuddy 里快速落地，我建议把这类任务统一抽象成下面这个结构：

1. 任务输入：描述要生成什么内容
2. Agent 执行：生成文件或站点目录
3. 发布判断：是单文件还是整站目录
4. 发布执行：上传并完成发布
5. 任务输出：返回可访问链接

如果是持续更新类任务，再补一条：

6. 更新已有站点，而不是每次新建一个入口

这样做的好处是，WorkBuddy 看到的是一个稳定的任务协议，Agent 也不需要每次临时决定怎么交付。

## 八、我现在怎么看这件事

做了一段时间之后，我越来越觉得，Agent 系统里真正影响体验的，不是“它能不能生成”，而是“它能不能把结果交出去”。

一旦发布链路打通，很多事情就顺了：

- 图片生成不再停留在本地文件
- 报告导出不再停留在临时目录
- 文档生成可以直接变成可访问页面
- 每日更新类任务也能稳定迭代

这其实也是我最近在 WorkBuddy 里最想补齐的一环：让 Agent 产生的结果，尽量在任务结束时就是可分享、可访问、可回看的。

## 九、附一个开源实现参考

如果你想看一个实际可运行的实现，我这次用到的是这个开源项目：

- GitHub: [https://github.com/okfilecom/okfile](https://github.com/okfilecom/okfile)

它比较适合这个场景的原因是：

- 文件和站点目录都能处理
- 可以直接返回可访问链接
- 图片、视频、PDF 这类内容适合直接预览
- 根目录没有 `index.html` 时可以展示目录列表
- 已发布站点后续可以继续更新

对我来说，这类实现更接近 Agent 工作流里的“最终交付层”。

如果你也在做 WorkBuddy、Agent 或自动化交付相关的流程，我很建议把“发布”这一步提前设计进去，而不是等到最后再补。
